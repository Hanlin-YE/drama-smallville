# Vibe Coding 新品发布舆情推演沙盘 PRD v1.0

> 日期：2026-06-12  
> 定位：面向黑客松团队、AI 原生开发者、独立开发者和开发者工具团队的新品发布舆情推演工作台  
> 核心原则：本质仍是舆情推演，但落地场景从“品牌危机舆情”切换为“新品发布舆情”；不做泛用预测引擎，不做流量玄学工具，不做舆论操纵工具。  
> 产品 Slogan：**发布前，先模拟一遍开发者世界会怎么反应。**

---

## 0. 当前产品判断

本项目的核心能力不是“预测未来”，而是：

> **围绕一个具体事件，模拟不同人群、平台和意见角色的反应路径，帮助用户提前识别传播机会、质疑点、采用阻力和策略调整空间。**

此前的“品牌危机 72 小时舆情推演”商业价值明确，但气质偏 B 端、公关和风控。若目标是黑客松比赛，**Vibe Coding 新品发布舆情推演**更贴近比赛主题，也更容易做出有画面感的 Demo。

因此，本版将落地场景调整为：

> **Vibe Coding 新品发布舆情推演沙盘。**

这个方向仍然属于舆情推演，只是舆情对象从“危机事件”变成“新品发布”：

| 危机舆情 | 新品发布舆情 |
|---|---|
| 负面事件 | 新品发布 |
| 品牌方 | 产品团队 / 黑客松团队 |
| 消费者 | 开发者用户 |
| 媒体 | 科技媒体 / Newsletter |
| KOL | AI 工具博主 / 开发者 KOL |
| 反对者 | 怀疑型开发者 / 竞品用户 |
| 支持者 | 早期粉丝 / 开源贡献者 |
| 平台算法 | GitHub / Product Hunt / X / HN 推荐机制 |
| 危机回应策略 | 发布策略 / README / Demo / 定价 / 开源策略 |
| 二次爆点 | 功能爆点 / 争议点 / 吐槽点 |
| 情绪曲线 | 兴趣 / 信任 / 质疑 / 试用意愿曲线 |

## 1. 产品一句话

一个面向 AI 编程产品、开发者工具和黑客松项目的交互式发布舆情推演沙盘：用户输入新品描述、目标用户、核心功能、发布渠道和发布材料，系统通过规则引擎、历史发布案例库和多 Agent 时间步推演，输出 0h / 6h / 24h / 72h / 7d 的传播路径、开发者反应、KOL 转发可能性、GitHub / Product Hunt / Hacker News 反馈、采用阻力、质疑点和发布优化建议。

## 2. 产品定位

### 2.1 我们不是增长预测工具

产品不承诺预测 GitHub star 数、Product Hunt 排名、转化率或真实用户增长。

产品定位是：

> **发布舆情推演 + 开发者反馈预演 + 发布策略优化。**

用户真正需要的不是一句“会不会火”，而是：

- 哪些人会第一批试用？
- 哪些人会质疑它只是 wrapper？
- 哪个功能最有传播钩子？
- README、Demo、发布帖哪里最容易劝退开发者？
- GitHub / Product Hunt / X / HN 上分别会出现什么反应？
- 发布前应该先补什么材料？
- 哪些质疑需要在发布帖里主动回应？

### 2.2 我们不是普通发布文案生成器

常见 AI 发布工具是：

```text
输入产品描述 -> 生成 Product Hunt 文案 / Tweet / README
```

本产品要做的是：

```text
输入新品发布信息 -> 结构化发布事件 -> 模拟不同开发者群体反应 -> 比较发布策略 -> 输出改版建议
```

文案只是副产物，真正的产品是可交互、可对比、可审计的发布舆情沙盘。

### 2.3 我们不是舆论操纵工具

产品禁止输出：

- 刷榜、刷 star、刷评论建议。
- 虚假用户评价。
- 伪造社区热度。
- 攻击竞品或误导用户。
- 规避平台规则的增长策略。

产品只提供：

- 发布材料诊断。
- 目标人群反应推演。
- 渠道优先级判断。
- 质疑点和采用阻力识别。
- 合规、真实、可执行的发布优化建议。

## 3. 第一落地场景

### 3.1 场景名称

**Vibe Coding 新品发布舆情推演沙盘**

### 3.2 核心问题

> 一个 Vibe Coding / AI 编程新品发布后，开发者社区、KOL、早期用户、怀疑者和竞品视角会如何反应？发布团队应该如何优化 Demo、README、发布帖、渠道选择和产品叙事？

### 3.3 典型用户

优先面向：

1. 黑客松参赛团队。
2. AI 原生开发者。
3. 独立开发者。
4. 开源项目维护者。
5. 开发者工具创业团队。
6. AI coding 工具插件作者。
7. 早期 SaaS 产品团队。

MVP 阶段建议先服务：

> **黑客松团队和 AI 原生独立开发者。**

原因：

- 与比赛主题高度契合。
- 他们天然需要发布、展示和包装产品。
- 他们愿意快速试用新工具。
- 输入材料容易获得：产品想法、README、Demo、发布帖草稿。
- 输出可以直接反哺参赛项目本身。

### 3.4 典型产品类型

MVP 可覆盖以下新品类型：

1. Vibe Coding 工具。
2. Cursor / VS Code / JetBrains 插件。
3. 多 Agent 开发 workflow。
4. AI PRD / 需求拆解 / 代码生成工具。
5. 开源开发者工具。
6. AI SaaS 小工具。
7. 黑客松 Demo 项目。

## 4. 核心体验

### 4.1 5 分钟体验

用户第一次打开产品时，应完成一次完整推演：

```text
输入产品：
我们做了一个 VS Code 插件，可以把自然语言需求转成多 Agent 开发任务，并自动生成 PRD、任务拆解和代码修改建议。

选择目标用户：
独立开发者 / 黑客松团队 / AI 工程师

选择发布渠道：
GitHub / Product Hunt / X / Hacker News

输入发布材料：
README 草稿、产品一句话、Demo 链接或功能列表

点击推演：
系统输出 0h / 6h / 24h / 72h / 7d 的发布舆情路径。
```

用户应该能看到：

- 哪些开发者群体最可能感兴趣。
- 哪些开发者会质疑或吐槽。
- 哪个渠道最适合首发。
- 哪个功能最有传播钩子。
- README 或发布帖中哪里不够可信。
- Demo 是否足够支撑传播。
- 发布前应该优先补哪些材料。

### 4.2 核心交互流程

```text
输入新品发布信息
-> 发布事件结构化
-> 选择目标人群和发布渠道
-> 运行多 Agent 推演
-> 查看 7 天传播路径
-> 对比不同发布策略
-> 生成发布优化建议
```

### 4.3 交互原则

1. 图表优先，报告后置。
2. 允许用户快速切换发布渠道、Demo 成熟度、是否开源、定价策略。
3. 每条关键结论都能点开看到依据、规则、案例或 Agent 观点。
4. 输出必须区分事实、假设、推理和建议。
5. 产品语言要贴近开发者，不用营销黑话。
6. 结果要能直接转化为 README、发布帖、Demo 脚本和路线图修改。

## 6. 输出体系

### 6.1 默认输出六件套

1. **7 天关注度曲线**  
   展示 0h / 6h / 24h / 72h / 7d 的关注趋势、峰值机会和衰减风险。

2. **兴趣 / 质疑 / 信任 / 试用意愿曲线**  
   展示开发者社区从“看到”到“相信”再到“愿意试用”的变化。

3. **渠道扩散路径**  
   展示 GitHub、Product Hunt、X、Hacker News、Reddit、Discord、微信群之间的传播迁移。

4. **开发者人群反应矩阵**  
   展示独立开发者、开源开发者、企业工程师、AI KOL、怀疑型 HN 用户、竞品用户的典型反应。

5. **爆点功能与吐槽点雷达**  
   标出最可能被传播的功能，以及最可能被质疑的问题。

6. **发布优化建议**  
   给出 README、Demo、发布帖、定价、开源策略、渠道顺序的具体修改建议。

### 6.2 策略对比输出

用户应能比较不同发布策略：

| 发布策略 | 关注度潜力 | 信任风险 | 试用转化 | 典型问题 | 建议 |
|---|---:|---:|---:|---|---|
| GitHub 首发 + 开源 | 高 | 中 | 中-高 | 需要清晰 README 和安装路径 | 适合开发者工具 |
| Product Hunt 首发 | 中-高 | 中 | 中 | 需要视觉资产和清晰 tagline | 适合面向广泛早期用户 |
| Hacker News 首发 | 高波动 | 高 | 中 | 容易被质疑夸大 AI | 必须具体、诚实、技术细节充分 |
| X KOL 扩散 | 中 | 中 | 中 | 依赖传播钩子 | 需要短视频或 GIF |
| 私域社群内测 | 中 | 低 | 高 | 增长较慢 | 适合先打磨反馈 |

### 6.3 输出边界

所有输出必须明确标注：

```text
这是基于当前输入、历史发布案例和规则假设的舆情推演结果，不代表真实增长预测。
```

## 7. 多 Agent 设计

### 7.1 MVP Agent

第一版建议使用 8 个 Agent：

| Agent | 代表对象 | 职责 |
|---|---|---|
| Launch Parser Agent | 发布分析员 | 提取产品类型、目标用户、功能、渠道、成熟度和发布目标 |
| Indie Dev Agent | 独立开发者 | 判断是否想试用、是否愿意收藏、是否觉得有用 |
| GitHub Dev Agent | 开源开发者 | 判断 README、安装门槛、star 动机和维护可信度 |
| HN Skeptic Agent | Hacker News 怀疑型用户 | 找同质化、夸大宣传、技术可信度和 wrapper 质疑 |
| AI KOL Agent | AI 工具传播者 | 判断是否有转发钩子、标题价值和内容素材 |
| Enterprise Engineer Agent | 企业工程师 | 判断隐私、安全、稳定性、集成和团队采用阻力 |
| Competitor Agent | 竞品视角 | 指出差异化不足、替代风险和定位漏洞 |
| Launch Judge Agent | 汇总裁判 | 聚合路径、识别机会和风险，生成发布优化建议 |

### 7.2 Agent 协作流程

```text
Round 1：Launch Parser Agent 输出结构化发布事件
Round 2：Indie Dev / GitHub Dev / HN Skeptic / AI KOL / Enterprise Engineer / Competitor 分别判断
Round 3：Launch Judge Agent 聚合分歧，比较不同发布策略
Round 4：生成 0h / 6h / 24h / 72h / 7d 推演结果
Round 5：输出发布材料修改建议、渠道优先级和风险清单
```

### 7.3 Agent 约束

1. Agent 必须按照 schema 输出。
2. Agent 不能编造已发生的真实增长数据。
3. Agent 必须区分事实、假设和推理。
4. Launch Judge Agent 不能输出刷榜、刷 star、虚假评论或攻击竞品建议。
5. 每个关键结论必须绑定规则、案例或明确假设。

## 8. 技术实现路线

### 8.1 推荐路线

第一阶段采用：

> **规则引擎 + LLM 多 Agent + 历史发布案例 RAG + 时间步 simulation。**

不建议第一版直接做全平台实时监控，也不建议一开始做复杂 Agent-Based Modeling。可以参考 MiroFish 的“seed material -> ontology / graph -> agents -> social simulation -> structured report”思想，但不要照搬“预测万物”的泛化定位。

### 8.2 方案 A：规则引擎 + LLM Agent

这是最快的 Demo 方案，也是 MVP 主体。

```text
新品发布输入
-> LaunchEventSchema 结构化
-> 规则引擎判断初始发布风险和机会
-> 多 Agent 分角色推演
-> 时间步模拟 0h / 6h / 24h / 72h / 7d
-> 输出曲线、质疑点、渠道策略和发布建议
```

核心能力：

- 发布信息结构化。
- 发布准备度评分。
- 开发者人群反应判断。
- 渠道策略对比。
- 7 天时间步输出。
- JSONL trace 记录。

优点：

- 2-4 周可做出可演示版本。
- 解释性强。
- 数据依赖较低。
- 适合黑客松演示。

缺点：

- 曲线是情景模拟，不是真实增长预测。
- 需要明确表达“推演边界”。

### 8.3 方案 B：历史发布案例 RAG + 类比推演

该方案用于提升可信度，建议和方案 A 组合。

```text
新品发布输入
-> 提取产品标签和发布标签
-> 检索历史相似发布案例
-> 对比案例中的传播路径、质疑点和爆点
-> 多 Agent 基于案例推演
-> 输出当前新品发布路径
```

案例库字段：

| 字段 | 示例 |
|---|---|
| 产品名称 | Cursor |
| 产品类型 | AI coding tool |
| 发布渠道 | X / GitHub / Product Hunt |
| 目标用户 | 开发者 / AI 工程师 |
| 核心钩子 | AI pair programmer |
| Demo 形式 | 视频 / 可试用产品 |
| 开源状态 | 非开源 / 部分开源 |
| 主要质疑 | 隐私、稳定性、是否替代 IDE |
| 爆点功能 | Tab completion、agent workflow |
| 传播峰值 | 发布后若干天内 |
| 后续转化 | waitlist、安装、付费、社区讨论 |

技术组件：

- Embedding 模型。
- Chroma / pgvector。
- Postgres。
- 案例标签体系。
- 相似案例解释器。

优点：

- 用户更容易信任。
- 能减少纯 LLM 胡编。
- 长期可形成发布案例壁垒。

缺点：

- 需要人工整理高质量案例。
- 案例标签标准需要持续迭代。

### 8.4 方案 C：Agent-Based Modeling

该方案适合第二阶段，不建议第一版重做。

```text
开发者群体 Agent
KOL Agent
开源维护者 Agent
怀疑型用户 Agent
企业工程师 Agent
平台推荐机制
-> 每个时间步模拟浏览、转发、star、试用、吐槽、二次传播
```

可模拟群体：

- 独立开发者。
- 开源开发者。
- 黑客松团队。
- AI 工具 KOL。
- 企业工程师。
- HN 怀疑型用户。
- 竞品用户。

典型参数：

- 点击概率。
- star 概率。
- 试用概率。
- 质疑概率。
- 转发概率。
- README 信任阈值。
- Demo 敏感度。
- 开源偏好。
- 隐私风险敏感度。
- 平台扩散系数。

优点：

- 沙盘感更强。
- 可视化更丰富。
- 更接近真正 simulation。

缺点：

- 参数校准困难。
- 无真实数据时容易显得伪科学。
- 开发周期更长。

### 8.5 方案 D：实时发布监听 + 推演

该方案适合商业化阶段。

```text
关键词 / GitHub repo / Product Hunt 页面 / X 话题监听
-> 抓取 star、issue、评论、转发、Newsletter 和社区讨论信号
-> 情绪与采用阻力分析
-> 触发二次推演
-> 输出发布优化和迭代建议
```

技术组件：

- GitHub API。
- Product Hunt API 或页面数据。
- Hacker News API。
- X / Reddit / Discord 数据源。
- 趋势检测。
- 情绪和质疑点分类模型。
- 飞书 / Slack / 邮件通知。

优点：

- 商业价值更高。
- 可从推演工具升级为发布作战台。
- 用户粘性更强。

缺点：

- 数据源和 API 限制较多。
- 平台规则和合规风险更高。
- 系统复杂度明显上升。

## 12. 历史发布案例库

### 12.1 MVP 案例数量

第一版建议人工整理 20-40 个高质量发布案例，而不是追求大量自动抓取。

案例优先级：

1. Cursor。
2. Windsurf。
3. Lovable。
4. Bolt。
5. v0。
6. Devin。
7. OpenHands。
8. Claude Code。
9. Codex。
10. 典型 Product Hunt 开发者工具发布。
11. 典型 Hacker News 开源工具发布。

### 12.2 案例使用方式

每次推演检索 3-5 个相似案例，用于：

- 给 Agent 提供类比背景。
- 支撑渠道传播判断。
- 展示“相似发布案例”。
- 提醒用户哪些定位、文案或 Demo 曾经造成质疑。

### 12.3 案例质量原则

1. 案例必须有明确发布时间线。
2. 案例必须区分事实数据与社区评论。
3. 案例必须标注发布渠道、核心钩子和主要质疑。
4. 案例必须保留来源链接或来源说明。
5. 案例不用于复制增长手段，只用于发布策略诊断和风险识别。

## 13. 合规与安全边界

### 13.1 产品边界

产品定位为：

> 新品发布舆情分析、开发者反馈推演和发布策略优化辅助工具。

产品不提供：

- 刷榜。
- 刷 star。
- 刷评论。
- 虚假用户评价。
- 攻击竞品。
- 伪造社区热度。
- 规避平台规则。
- 恶意操纵公众意见。

### 13.2 输出安全要求

所有策略建议必须遵守：

1. 不鼓励虚假陈述。
2. 不鼓励攻击个人、群体或竞品。
3. 不鼓励诱导平台算法。
4. 不鼓励伪造用户反馈。
5. 不鼓励制造虚假稀缺或虚假热度。
6. 优先建议真实 Demo、清晰文档、透明限制和诚实发布。

### 13.3 风险提示

每份报告底部必须包含：

```text
本结果为基于当前输入、历史发布案例和规则假设生成的舆情推演，不代表真实增长预测，不应作为唯一发布决策依据。实际发布需结合产品成熟度、用户反馈、渠道规则和团队资源判断。
```

## 15. 验收标准

MVP 通过标准：

1. 用户可以输入一个 Vibe Coding / 开发者工具新品。
2. 系统可以结构化提取产品类型、目标用户、核心功能、渠道、Demo 成熟度和发布目标。
3. 用户可以选择至少 5 种发布策略或渠道组合。
4. 系统可以在 30-60 秒内完成一次推演。
5. 输出包含 0h / 6h / 24h / 72h / 7d 的关注度变化。
6. 输出包含兴趣、质疑、信任和试用意愿变化。
7. 输出包含至少 5 类开发者人群反应。
8. 输出包含爆点功能、吐槽点和反转条件。
9. 输出可以比较不同发布策略。
10. 每个关键结论至少绑定一个规则、案例或明确假设。
11. 输出不包含刷榜、刷 star、虚假评论、攻击竞品或规避平台规则建议。
12. 同一输入重复运行时结构稳定，不应完全发散。

## 16. 标准测试情景

MVP 应准备 5 个标准测试情景：

### 16.1 VS Code Vibe Coding 插件

一个 VS Code 插件，可以把自然语言需求拆成多 Agent 开发任务，并生成 PRD、任务列表和代码修改建议。

测试重点：

- 是否有足够清晰的差异化。
- 开发者是否会质疑只是 ChatGPT wrapper。
- Demo GIF 是否能显著提升信任。

### 16.2 GitHub 开源 Agent Workflow

一个开源多 Agent workflow，可以自动把 issue 转成实现计划、测试计划和 PR 修改建议。

测试重点：

- GitHub README 是否足够可信。
- 开源是否提升 star 动机。
- 安装复杂度是否降低试用意愿。

### 16.3 黑客松 AI SaaS Demo

一个黑客松团队做的 AI SaaS，可以根据一句需求生成落地页、后端接口和数据库 schema。

测试重点：

- Product Hunt 和 X 的传播钩子。
- Demo 成熟度对转化的影响。
- 是否容易被认为同质化。

### 16.4 企业代码助手

一个面向企业团队的代码助手，强调隐私、安全和团队知识库集成。

测试重点：

- 企业工程师的隐私和安全顾虑。
- 云端处理是否造成信任阻力。
- 文档是否需要更强调部署模式。

### 16.5 HN 发布帖

一个 AI 编程工具准备在 Hacker News 发布，文案中出现“fully autonomous developer”和“replace engineers”等表述。

测试重点：

- 夸大 AI 能力是否触发反感。
- 是否需要主动承认限制。
- 技术细节是否能缓解质疑。

## 18. PMF 验证

早期不要问：

> 你觉得它预测准不准？

应该问：

> 这个工具有没有帮你提前看到某个被忽略的发布阻力或传播机会？

正向信号：

1. 用户会反复切换发布渠道和 Demo 成熟度。
2. 用户主动修改 README、tagline 或发布帖。
3. 用户要求加入自己的竞品和案例。
4. 用户希望导出发布诊断报告。
5. 用户希望接入 GitHub / Product Hunt / HN 真实反馈。
6. 用户愿意在正式发布前使用它做检查。

负向信号：

1. 用户只把它当普通文案生成器。
2. 用户不相信曲线和评分。
3. 用户认为输出没有超过一个资深开发者朋友的建议。
4. 用户不愿输入真实产品材料。
5. 用户无法说明它相比人工 brainstorming 的增量价值。
6. 用户只关注文案，不关心模拟过程。
